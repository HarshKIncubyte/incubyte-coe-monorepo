# Three Laws of TDD

## Key Notes

* **First Law:** Do not write production code unless it is required to make a failing unit test pass.
* **Second Law:** Write only enough of a unit test to make it fail. A compilation error is also considered a failure.
* **Third Law:** Write only enough production code to make the failing test pass.
* Work on **one failing test at a time**. Avoid writing multiple failing tests before implementing the solution.
* Keep both the test and production code as small as possible in each cycle.
* The three laws enforce small development steps, provide fast feedback, and naturally lead to the **Red → Green → Refactor** cycle.
* The goal of TDD is not just testing the code but using tests to drive the design of the software.
